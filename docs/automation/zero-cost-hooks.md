# Zero-cost hooks and automation guardrails

## Purpose

Lamezia Trasparente Monitor uses repository-level hooks to make agentic development reviewable, auditable and economically safe. The first implementation deliberately stays inside GitHub Actions on a public repository and uses only standard `ubuntu-latest` runners.

The hooks are not a substitute for human review. They create a controlled workflow around Codex or other coding agents by checking that a pull request is traceable, does not weaken civic safeguards, does not introduce obvious repository hygiene problems and does not move the project towards paid automation.

## Zero-spend policy

Allowed:

- standard GitHub-hosted `ubuntu-latest` runners;
- pull-request checks;
- push-to-`main` checks;
- lightweight Node scripts committed in the repository;
- optional manual `workflow_dispatch` runs;
- short-lived, small logs generated by the workflow itself.

Forbidden unless Giovanni explicitly changes the policy:

- larger runners;
- GPU runners;
- self-hosted runners used as paid infrastructure;
- paid external monitoring services;
- Codespaces prebuilds;
- long-retention or heavy artifacts;
- high-frequency scheduled workflows introduced without a specific civic-operational reason.

The current policy intentionally requires executable workflows to use `runs-on: ubuntu-latest`. This is stricter than generic GitHub Actions availability because the project objective is operational simplicity and zero avoidable cost.

## Implemented hooks

### PR hook guard

Workflow: `.github/workflows/hook-guard.yml`  
Script: `scripts/hooks/pr-guard.mjs`

Runs on pull requests and checks:

- changed files;
- unresolved conflict markers;
- high-confidence secret patterns;
- workflow runner policy;
- whether the PR references an issue;
- whether the PR body contains a Human Decision Ledger;
- whether source/limitations and civic-safeguard checks are documented;
- whether civic content/data/source-impact paths appear to have been touched.

Hard blockers fail the workflow. Advisory items are reported as warnings during the v0 phase.

### Hook ledger comment

Workflow: `.github/workflows/hook-guard.yml`  
Script: `scripts/hooks/comment-ledger.mjs`

After every PR hook run, the workflow creates or updates a single PR comment marked as:

```text
<!-- ltm-hook-guard -->
```

The comment reports changed files, blockers, warnings, notes and a compact Human Decision Ledger. This avoids repeated manual reconstruction of what Giovanni has to do next.

### Zero-cost automation policy check

Workflow: `.github/workflows/zero-cost-policy.yml`  
Script: `scripts/hooks/zero-cost-policy-check.mjs`

Runs when workflow files or the policy scripts change. It fails if executable workflow files use anything other than `runs-on: ubuntu-latest`.

### Deploy smoke check

Workflow: `.github/workflows/deploy-smoke.yml`

Runs after relevant pushes to `main` and can also be launched manually. It builds the v0 static fallback and runs the provider-neutral smoke check. If the repository variable `PUBLIC_SITE_URL` is set, or if a manual `public_url` input is supplied, it also performs a lightweight live URL check.

The live URL check is deliberately optional. Absence of `PUBLIC_SITE_URL` is not a failure because the repository must remain deploy-provider neutral.

## Operating model

The intended sequence is:

```text
Issue
↓
Codex or human implementation
↓
Pull request
↓
PR hook guard
↓
Hook ledger comment
↓
Human review
↓
Merge
↓
Deploy smoke check
```

A PR should not be treated as ready merely because code was generated. It is ready only when the pull request is reviewable, has clear validation evidence, preserves civic safeguards and does not introduce cost-bearing automation.

## Civic-source posture

The hooks support a deliberately stronger scouting posture while keeping the publication layer prudent.

The project may generate public-interest leads, red-flag framings, FOIA/accesso civico prompts, comparative benchmarks and editorially useful dossiers. Public-facing publication must still distinguish:

- facts;
- allegations;
- missing data;
- interpretations;
- legal risk;
- human-gated decisions.

The hook checks cannot verify factual correctness. They only force visibility of source and limitation handling so that risky material is not merged silently.
